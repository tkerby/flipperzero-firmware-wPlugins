# FAP (пакет приложений Flipper) {#apps_on_sd_card}

[fbt](./fbt.md) поддерживает сборку приложений как файлов FAP. FAP — это по сути исполняемые файлы `.elf` с дополнительными метаданными и ресурсами.

FAP собираются целью `faps`. Также их можно развернуть в папку `dist` целью `fap_dist`.

FAP не зависят от запуска на конкретной версии прошивки. Совместимость определяется метаданными FAP, которые включают требуемую [версию API](#api-versioning).

## Как настроить приложение для сборки как FAP {#fap-howto}

FAP создаются и разрабатываются так же, как внутренние приложения, являющиеся частью прошивки.

Чтобы собрать приложение как FAP, создайте папку с исходным кодом приложения в `applications_user`, затем напишите код так, как вы бы это сделали при создании обычного встроенного приложения. Затем настройте его манифест `application.fam`, установив `apptype` в `FlipperAppType.EXTERNAL`. Подробнее см. [манифесты приложений Flipper](AppManifests.md).

- Для сборки приложения выполните `./fbt fap_{APPID}`, где APPID — ID вашего приложения в его манифесте.
- Для сборки приложения и его загрузки по USB для запуска на Flipper используйте `./fbt launch APPSRC=applications_user/path/to/app`. Эта команда настроена в профиле [VS Code по умолчанию](../.vscode/ReadMe.md) как действие сборки "Launch App on Flipper" (меню Ctrl+Shift+B).
- Для сборки приложения без загрузки на Flipper используйте `./fbt build APPSRC=applications_user/path/to/app`. Эта команда также доступна в конфигурации VSCode как "Build App".
- Для сборки всех FAP выполните `./fbt faps` или `./fbt fap_dist`.

## Ресурсы FAP

FAP могут включать статические и анимированные изображения как приватные ресурсы. Они будут автоматически скомпилированы вместе с исходниками приложения и могут быть использованы так же, как ресурсы основной прошивки.

Для использования этой функции поместите изображения в подпапку внутри папки приложения, затем укажите эту папку в манифесте приложения в поле `fap_icon_assets`. Подробнее см. [манифесты приложений Flipper](AppManifests.md).

Для использования этих ресурсов в приложении добавьте `#include "{APPID}_icons.h"` в исходный код приложения, где `{APPID}` — значение поля `appid` из манифеста вашего приложения. Затем вы можете использовать все иконки из ресурсов вашего приложения так же, как если бы они были частью `assets_icons.h` основной прошивки.

Изображения и анимированные иконки должны следовать тем же [соглашениям об именовании](../assets/ReadMe.md), что и в основной прошивке.

## Отладка FAP

`fbt` включает скрипт для gdb-py для обеспечения поддержки отладки FAP — `debug/flipperapps.py`. Он загружается в конфигурациях отладки по умолчанию через `fbt` и стандартные конфигурации VS Code.

С его помощью вы можете отлаживать FAP так, как если бы они были частью основной прошивки — исследовать переменные, устанавливать точки останова, пошагово выполнять код и т.д.

Если сессия отладки активна, прошивка вызовет точку останова после загрузки FAP в память, но до выполнения любого кода из него. Это позволяет устанавливать точки останова в коде FAP. Обратите внимание, что любые точки останова, установленные до загрузки FAP, могут потребовать повторной установки после фактической загрузки FAP, поскольку отладчик не может знать точный адрес кода FAP до его загрузки.

### Настройка среды отладки

Скрипт поддержки отладки ищет отладочную информацию в последней папке сборки прошивки (`build/latest`). В эту папку `fbt` создаёт симлинк на последнюю конфигурацию сборки (Debug или Release) при запуске `./fbt` для выбранной конфигурации. Подробнее см. [документацию fbt](fbt.md).

Для отладки FAP выполните следующее:

1. Соберите прошивку с помощью `./fbt`
2. Прошейте с помощью `./fbt flash`
3. [Соберите ваш FAP](#fap-howto) и запустите его на Flipper

После этого вы можете подключить отладчик к целевому MCU с помощью `./fbt debug` или VS Code и использовать все функции отладки.

**Важно**, чтобы тип сборки прошивки и приложения (debug/release) совпадали и чтобы соответствующая папка прошивки была связана как `build/latest`. В противном случае отладка не будет работать.

## Как Flipper запускает приложение с SD-карты

MCU Flipper не может выполнять код напрямую с внешнего хранилища, поэтому его нужно сначала скопировать в ОЗУ. Это делает App Loader, отвечающий за загрузку FAP с SD-карты, проверку его целостности и совместимости, копирование в ОЗУ и корректировку для нового расположения.

Поскольку FAP должен быть загружен в ОЗУ для выполнения, объём ОЗУ, доступный для выделения из кучи, уменьшается по сравнению с запуском того же приложения из флеш-памяти как части прошивки. Обратите внимание, что объём занятой ОЗУ меньше общего размера файла FAP, поскольку выделяются только секции кода и данных, в то время как файл FAP включает дополнительную информацию, используемую только во время загрузки приложения.

Приложения собираются для конкретной версии API. Это часть определения аппаратной цели и содержит мажорную и минорную версии. App Loader проверяет, совпадает ли мажорная версия API приложения с мажорной версией API прошивки.

App Loader выделяет память для приложения и копирует его в ОЗУ, обрабатывая перемещения и предоставляя конкретные адреса для импортируемых символов с использованием [таблицы символов](#symbol-table). Затем он запускает приложение.

## Версионирование API {#api-versioning}

Не все части прошивки доступны для внешних приложений. Подмножество доступных функций и переменных определено в файле "api_symbols.csv", который является частью определения цели прошивки в каталоге `targets/`.

`fbt` использует семантическое версионирование для API. Мажорная версия увеличивается при наличии критических изменений в API. Минорная версия увеличивается при добавлении новых функций.

Критические изменения включают:

- Удаление функции или глобальной переменной
- Изменение сигнатуры функции

Версионирование API в основном автоматизировано `fbt`. При пересборке прошивки `fbt` проверяет, есть ли изменения в API, экспортируемом заголовками из `SDK_HEADERS`. Если да, он останавливает сборку, корректирует версию API и просит пользователя просмотреть изменения в файле `.csv`. Новые записи помечаются знаком "`?`", и пользователь должен изменить знак на "`+`" для экспорта записи в FAP или на "`-`" для её недоступности.

`fbt` не позволит собрать прошивку, пока все записи "`?`" не будут изменены на "`+`" или "`-`".

**Важно:** `fbt` автоматически управляет версией API. Единственный случай, когда разрешено (и требуется) ручное увеличение мажорной версии API — когда существующие записи "`+`" должны быть изменены на "`-`".

### Таблица символов {#symbol-table}

Таблица символов — это список символов, экспортируемых прошивкой и доступных для внешних приложений. Она генерируется `fbt` из файла символов API и используется App Loader для разрешения адресов импортируемых символов. Собирается как часть приложения `fap_loader`.

`fbt` также проверяет, все ли импортируемые символы присутствуют в таблице символов. Если есть отсутствующие символы, он выдаст предупреждение с их перечнем. Приложение не сможет работать на устройстве, пока все требуемые символы не будут предоставлены в таблице символов.
